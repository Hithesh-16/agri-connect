Critical gaps (blockers to production)

No payment gateway implemented Blocker
Razorpay listed in architecture plans and Phase 3 roadmap but has 0% implementation. Without this, the entire marketplace monetisation is non-functional. Escrow, UPI, and payout flows are all missing.
→ Implement Razorpay before any vendor goes live. Route API for escrow is non-trivial — needs dedicated 2-week sprint.
Mobile app not wired to backend Blocker
React Native 0.81 app exists but is not connected to the 17 backend routes. TanStack Query is listed in the stack but no API calls are implemented. Vision Camera is listed but no ML inference pipeline exists.
→ Wire API calls before adding any new mobile features. Prioritise auth → prices → booking → notifications.
No real-time update mechanism Critical
Booking status changes (CONFIRMED → IN_PROGRESS → COMPLETED) require the farmer and vendor to see updates live. No WebSocket or SSE layer exists. Polling would destroy performance at scale.
→ Add Socket.io or Ably for booking room events. SSE is simpler for one-way notifications.
No async job queue Critical
SMS, WhatsApp, and push notifications are fire-and-forget in the current design. During peak harvest bookings (100s/minute), synchronous notification calls will block API responses and cause timeouts. Price ingestion from AGMARKNET also needs a queue.
→ Add BullMQ + Redis. Notification workers dequeue asynchronously. Prevents API latency spikes.
No caching layer Critical
Mandi prices are fetched from AGMARKNET and weather from IMD/OpenWeather on every request. These rarely change within minutes. Under load, this will hit external API rate limits and add 200–800ms latency to every dashboard load.
→ Redis cache for prices (TTL 15 min), weather (TTL 30 min), scheme data (TTL 24h). Cache-aside pattern.
PostgreSQL on Neon — connection limit problem Critical
Neon's serverless PostgreSQL has connection limits (20–100 depending on plan). Express.js with Prisma holds persistent connections. At 50 concurrent users, you will hit connection exhaustion and get "too many connections" errors.
→ Add PgBouncer or use Neon's built-in connection pooling mode. Set Prisma connection_limit=5 per instance, pool externally.
No geo-search infrastructure Critical
The vendor discovery feature ("services near me within 5/10/25 km") is the core UX of the marketplace. The current PostgreSQL schema has no PostGIS extension, no spatial index, and no geospatial query. This is the #1 query in the whole app.
→ Enable PostGIS on Neon. Add GEOGRAPHY columns to ServiceProvider. Create GiST index. Use ST_DWithin() for radius queries.
No offline-first implementation on mobile Critical
WatermelonDB is listed in the architecture section but NOT in the current mobile stack. The document acknowledges rural users have intermittent 2G, but the app currently requires network for every interaction.
→ Add WatermelonDB to the mobile stack now. Cache prices, vendor listings, and booking data. Sync queue on reconnect.
No observability / monitoring Critical
No error tracking, no APM, no structured logging. When the service goes down at 6AM during harvest season, there's no alert, no trace, no log to debug. This is unacceptable for a production system.
→ Add Sentry (errors), Pino (structured logs), and uptime monitoring (BetterStack or UptimeRobot). Free tiers sufficient to start.

Document inconsistencies

OpenWeatherMap vs IMD — conflicting sources
Section 04 (current stack) lists OpenWeatherMap as an active service. Section 09 (architecture) and Section 08 (edge cases) list IMD as the target. IMD requires a formal licence agreement — it's not a simple API key signup. No migration path is documented.
→ Keep OpenWeatherMap for MVP. Plan IMD licensing in Phase 5. Document both clearly.
Decimal fields missing precision in Prisma models
All financial fields (totalAmount, platformFee, pricePerUnit, dailyRate) use Decimal type without specifying precision/scale. Prisma defaults to (65, 30) which is fine for storage but PostgreSQL aggregations on unbounded Decimal are slower than Decimal(10,2) or Decimal(15,4).
→ Specify @db.Decimal(15, 4) for currency fields, @db.Decimal(10, 2) for user-facing prices.
Mapbox GL listed as "plan" but Sentinel Hub also "plan" with no timeline
Both Mapbox GL (geo visualisation) and Sentinel Hub (satellite NDVI) appear in the current tech stack as planned. Sentinel Hub has a minimum plan cost of ~$25/month and requires crop-polygon data which doesn't exist yet. It's a Phase 7 feature but listed as current.
→ Move both to Phase 7. Use free Mapbox free tier for initial map features. Remove from "current stack" table.
TimescaleDB listed in architecture but no migration strategy
Price history data is currently in main PostgreSQL. The architecture plan lists TimescaleDB as a separate data store. Moving time-series data mid-production to a separate instance is a significant migration with dual-write complexity.
→ Start with TimescaleDB extension on the same PostgreSQL instance. Only separate to a standalone instance after 6 months of data accumulation.
Revenue projection assumes 10K bookings in Year 1 at INR 50L — math gap
10,000 bookings × avg INR 1,000 booking value × 5% effective commission (blended) = INR 5L, not INR 50L. To reach INR 50L from commissions alone needs either 100K bookings or 15% commission on INR 3,333 avg booking value.
→ Revise to include vendor subscription revenue + wallet fees. Year 1: 500 Pro vendors × INR 3,588/yr = INR 18L + 10K bookings × INR 150 avg commission = INR 15L = INR 33L. Still below 50L — revise down to INR 25L or revise assumptions up.
KYC flow mentions DigiLocker and UIDAI API but no implementation path
Aadhaar-based eKYC requires an AUA (Authentication User Agency) or KUA licence from UIDAI, which takes 3–6 months to obtain. DigiLocker API also needs formal agreement. The roadmap puts KYC in Phase 1 (Weeks 1-4) — this is unrealistic.
→ Phase 1: manual KYC (upload Aadhaar photo + selfie, human review). Phase 3: integrate KARZA or IDfy (licensed eKYC providers with API access). Phase 6+: direct UIDAI integration after licence.
Bhashini API described as "free" but requires ULCA registration
Bhashini (ULCA) is government-backed but requires developer registration and API key. The ASR quality for agricultural terms (crop names, pesticide names in regional languages) is currently poor. A production voice system needs fallback to a commercial ASR provider.
→ Register on ULCA now (free). For production, use Bhashini for translation/TTS, but Sarvam AI ASR or Google Cloud Speech for actual voice input until Bhashini ASR matures.
Section 12 monetisation mentions TDS handling (Section 194-O) but no implementation detail
Under Section 194-O, platforms deduct 1% TDS on payouts to vendors above INR 5L/year. This requires PAN collection, TDS computation, Form 26Q quarterly filing, and TDS certificate issuance. Not mentioned in the data model or roadmap.
→ Add TaxProfile model (PAN, GST, TDS threshold tracking). Integrate Razorpay's tax compliance tools. Add to Phase 3 scope.

Missing systems (not mentioned, but required)

OTA Updates for Mobile
With React Native + rural users on Play Store slow-update behaviour, bug fixes can't wait for a 3-day review cycle during harvest season. Need Expo EAS Updates or Microsoft CodePush for instant OTA deploys.
Image CDN + Compression
Vendor profiles and service listings will have photos. S3 raw upload without a CDN means 3–8MB images loading over 2G. Need Cloudflare R2 + Images (or AWS S3 + CloudFront) with WebP conversion and responsive sizing.
Fraud Detection
Fake reviews, fake check-ins (GPS spoofing for attendance), duplicate vendor accounts — all mentioned as risks but no technical solution listed. Need IP velocity checks, device fingerprinting, GPS polygon validation.
Refresh Token Rotation
Current auth is JWT only. JWTs expire but there's no refresh token rotation. A stolen JWT is valid until expiry with no revocation. Need short-lived access tokens (15 min) + long-lived refresh tokens stored in database.
Admin Panel (only 30% done)
KYC review, dispute resolution, vendor ban, price management, booking interventions — all need an admin interface. At 30% this is not usable. Dispute resolution (Phase 5) can't launch without admin tools.
CI/CD Pipeline
No deployment automation mentioned. Manual deploys at 3AM when the booking service crashes are not viable. Need GitHub Actions → automated testing → staging → production with rollback.
